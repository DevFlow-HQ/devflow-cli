# DevFlow-Owned Dumb Execution Loop With Marker-Only Termination

The execution stage runs the open issues through the selected provider with a deliberately *dumb loop* owned by DevFlow: DevFlow sequences provider turns but never parses, orders, filters, names, or moves issues. Each **Execution iteration** is a fresh dedicated managed session that receives only the content of the open `issues/*.md` files and the last few git commits — not the PRD body, project context, or other run artifacts; if the provider needs more it reads it itself. Within an iteration the provider selects a single `AFK` issue, completes it, moves the finished issue into `issues/done/`, and then commits if git changes are permitted; HITL issues are left untouched. The move — not the commit — is the cross-session memory: DevFlow injects only `issues/*.md`, never `issues/done/`, so a moved issue vanishes from the next fresh session and cannot be re-picked. The move deliberately precedes and is independent of the commit because users frequently deny the provider git permission; a moved-but-uncommitted issue still has its work in the working tree and the issue is already out of rotation, so no rework occurs, and `execution.json` truthfully records HEAD unchanged for that iteration. Sessions are intentionally one-issue-per-fresh-session because a single long session accumulates context until the provider degrades — bounding context per iteration is the whole point of looping rather than handing the provider everything at once.

Termination depends solely on the provider emitting a DevFlow-defined **no-more-tasks (terminal) marker**, which is the authoritative signal that no AFK work remains. An iteration that handles one issue emits a separate **iteration-complete marker** and DevFlow loops again. Two defensive exits supplement the terminal marker: an empty active issues directory (the provider finished everything but never signaled — checked as an absolute "zero `.md` files" condition, not a before/after diff) ends the run as a success, and a fixed per-run cap of `2N + 5` iterations (where `N` is the issue-file count captured once at stage start and never recomputed) ends it as a loud escalation. A session that ends without emitting either marker throws the existing `IncompleteProviderSessionError` and surfaces as a stage failure. There is no whole-phase retry and no in-session repair for this stage: a "retry" would just be another iteration grabbing whatever issue comes next, so it carries no meaning here, and with no progress guard there is nothing for a repair turn to fix. We deliberately removed an earlier no-progress guard that diffed the issues directory before/after each iteration because legitimate runs diverge from "directory shrank" in too many ways (the user stops the provider mid-turn, rearranges issues, or a move lands late), so the diff produced false failures; the marker contract plus the absolute empty-directory check are the only honest signals.

DevFlow records what it can observe truthfully *without parsing issue contents* in a per-iteration `execution.json` **Execution record**: for each iteration, which marker fired, the provider session id, and the git HEAD before→after (so committed work is captured from git ground truth rather than the provider's claims), plus a final stop reason (`terminal`/`no-file`/`cap`/`error`) and the completed-vs-remaining issue filenames. The record is explicitly per-iteration, not per-issue: DevFlow does not know which issue an iteration handled and will not invent that mapping. Validation of `execution.json` is lightweight (well-formed, at least one iteration), mirroring the existence-only philosophy of ADR-0009.

We rejected the alternative of DevFlow enumerating, parsing, and sequencing issues, running one managed session per named issue, and recording a per-issue success summary, because it contradicts DevFlow's delegator role and ADR-0009's provider-authored, existence-only issue contract, would require tightening the issue contract (slug/heading/blocked-by parsing) that ADR-0009 deliberately deferred, and would force DevFlow to assert per-issue success it cannot truthfully observe. The dumb loop keeps issue identity and selection entirely with the provider, keeps ADR-0009 intact, and matches the manually validated shell-script flow (dump all open issues plus recent commits into one provider turn, let the provider pick and do exactly one, repeat) that this stage automates. Tighten any of this — parse issues, add a progress guard, record per-issue — only when a real consumer truthfully requires it.
