# Adapter Diagnostic Tracing Is Metadata-Only And Boundary-Local

Adapter-deep `debug` tracing — provider events, marker scans, data-plane/tier resolution, phase transitions, the JSONL session-log locator, the hook socket, and PTY spawn/exit — is wired through the injected `Logger` but is deliberately constrained on two axes that the rest of DevFlow's diagnostic logging is not. First, adapter trace lines are **metadata-only**: they may carry event `type`, `source`, `structured`, `phaseId`, `origin`, `exitCode`/`signal`, a `messageLength` integer, the opaque DevFlow-injected marker token, and resolved capabilities — but never the body of a `submitted-user-message`, a `turn-completed.assistantMessage`, or the prompt argument passed to a provider executable. Second, adapter trace lines correlate by **provider + phaseId only**: adapters never learn the `runId` or `stage`, so those fields stay `null`/absent on adapter lines, and a maintainer locates the owning run by time-ordering adapter lines against the orchestrator's bracketing `info "stage started"`/`"stage completed"` entries, which already carry `runId` and `stage`.

We chose metadata-only because the **Diagnostic log** is durable on disk for thirty days and, unlike the **Grill transcript**, has no content policy — no marker stripping, no origin filtering, no redaction. Logging raw message bodies would create a second, unpoliced copy of everything the user and model said, including any secret a user pastes during the grill, directly contradicting the transcript-policy discipline that ADR-0004 keeps in orchestration. `messageLength` confirms content presence and shape for flow-debugging without copying the content; when a maintainer genuinely needs the text, the grill transcript and provider-native artifacts already hold it under their proper policies. We chose provider+phaseId-only correlation because threading `runId`/`stage` into the adapter contract would leak orchestration vocabulary across the provider boundary that ADR-0002 and ADR-0003 work to keep narrow — and it is unnecessary while DevFlow runs sessions strictly sequentially in one process, where timestamp interleaving already disambiguates. Marker-scan tracing is additionally kept turn-boundary-shaped (O(turns)/O(session)), never per-chunk (O(chunks)), so a single PTY session cannot flood the daily file; the per-chunk PTY scan stays silent until it matches, and only structured providers (via the phase manager) additionally log a turn-completed that carried no expected marker, since that "provider finished a turn but never signaled done" miss is the load-bearing diagnostic for a stuck stage.

A consequence worth stating: "fallback-tier selection" is traced as a **static resolution at adapter construction**, not a runtime fallback event, because no runtime hooks→JSONL degradation exists today (there is no automatic hook-failure relaunch). When that degradation is later built, it must emit its own `warn`-level line — a recoverable downgrade is `warn`, not `debug` — rather than being retrofitted onto this construction-time `debug` line. The two constraints here are forward policy: every future adapter trace must stay metadata-only and boundary-local. Revisit the correlation constraint only if DevFlow ever runs concurrent managed sessions in one process (a post-MVP parallelization concern that would reopen the event model anyway); revisit metadata-only never — promoting content into a durable, unpoliced log is the outcome this decision exists to prevent.

We rejected logging truncated message content (still copies user/model text into an unpoliced durable store, just less of it), threading `runId`/`stage` down for richer correlation (couples the provider boundary to orchestration concepts for the sole benefit of a log field that time-ordering already supplies), and per-chunk marker-scan tracing (drowns the signal it is meant to provide and churns the retained file).
