# Adapter Diagnostic Tracing Is Metadata-Only And Boundary-Local — Except Verbatim Hook Payloads

Adapter-deep `debug` tracing — provider events, marker scans, data-plane/tier resolution, phase transitions, the JSONL session-log locator, the hook socket, and PTY spawn/exit — is wired through the injected `Logger` and is constrained on two axes that the rest of DevFlow's diagnostic logging is not, **with one deliberate exception for raw provider-hook payloads** described below. First, adapter trace lines are **metadata-only by default**: they may carry event `type`, `source`, `structured`, `phaseId`, `origin`, `exitCode`/`signal`, a `messageLength` integer, the opaque DevFlow-injected marker token, and resolved capabilities — but never the body of a `submitted-user-message`, a `turn-completed.assistantMessage`, or the prompt argument passed to a provider executable. Second, adapter trace lines correlate by **provider + phaseId only**: adapters never learn the `runId` or `stage`, so those fields stay `null`/absent on adapter lines, and a maintainer locates the owning run by time-ordering adapter lines against the orchestrator's bracketing `info "stage started"`/`"stage completed"` entries, which already carry `runId` and `stage`.

## The hook-payload exception

When a **Provider hook** hands its JSON payload back to DevFlow over the **Hook IPC endpoint**, the *entire raw payload is logged verbatim* to the **Diagnostic log** at `debug` — not the `payloadLength`-only line the original version of this decision mandated. The justification is an asymmetry the original decision did not weigh: **a hook is consumed and gone**. Unlike a **JSONL session log**, which the provider persists to disk independently and DevFlow can re-read at any time, a hook payload exists only in the instant it crosses the socket. Once normalized and consumed it leaves no trace, so when a managed session behaves unexpectedly there is otherwise *no record at all* of what the provider actually sent — what event arrived, in what shape, with what content, in what order. That is precisely the situation in which debugging matters most, and `messageLength` alone cannot reconstruct it. Hooks arrive as JSON, so logging them verbatim is cheap and lossless.

The exception is **hook-only and narrow**. JSONL content is *not* additionally logged, because it is already durably persisted by the provider — re-reading the session log recovers it. PTY output is not logged. Every *other* adapter trace line stays metadata-only: only the raw hook-payload-received trace carries a body. The metadata-only default therefore still governs everything except this one ephemeral, otherwise-unrecoverable source.

## The accepted trade-off

The original decision rejected logging message bodies because the **Diagnostic log** is durable on disk for thirty days and, unlike the **Grill transcript**, has no content policy — no marker stripping, no origin filtering, no redaction — so logging bodies would create a second, unpoliced copy of everything the user and model said, **including any secret a user pastes during the grill**. That risk is real and we are *consciously accepting it* for hook payloads, because hook payloads carry the grill's submitted-user-messages and assistant content, so a pasted secret can now land in the diagnostic log. We accept it because (a) without the log there is no record of an ephemeral source at all, which we judge the larger operational risk, and (b) two existing mitigations bound the exposure: the diagnostic log lives under the repo-local `.devflow/` tree, which DevFlow adds to the target repository's `.gitignore`, so it never enters version control or leaves the machine, and the existing thirty-day prune still bounds how long it survives. We log payloads **verbatim** rather than with partial credential redaction, because best-effort field redaction is both unreliable and defeats the lossless-record purpose that justifies the exception in the first place.

## Unchanged constraints

The **boundary-local** axis is untouched: adapters still never learn `runId`/`stage`; time-ordering against the orchestrator's bracketing entries still supplies correlation; threading orchestration vocabulary across the provider boundary remains rejected for the reasons ADR-0002 and ADR-0003 give, and is unnecessary while DevFlow runs sessions strictly sequentially in one process. Marker-scan tracing stays turn-boundary-shaped (O(turns)/O(session)), never per-chunk (O(chunks)), so a single PTY session cannot flood the daily file; the per-chunk PTY scan stays silent until it matches, and only structured providers (via the phase manager) additionally log a turn-completed that carried no expected marker, since that "provider finished a turn but never signaled done" miss is the load-bearing diagnostic for a stuck stage.

"Fallback-tier selection" is still traced as a **static resolution at adapter construction**, not a runtime fallback event, because no runtime hooks→JSONL degradation exists today. When that degradation is later built it must emit its own `warn`-level line — a recoverable downgrade is `warn`, not `debug`.

## Forward policy

Metadata-only remains the default for adapter tracing, and the **hook-payload exception is the only sanctioned body in the diagnostic log** — do not extend verbatim-content logging to JSONL (already persisted), PTY, or any other source without a decision that re-weighs the same secret-exposure trade-off. Revisit the boundary-local constraint only if DevFlow ever runs concurrent managed sessions in one process (a post-MVP parallelization concern that would reopen the event model anyway). If the secret-exposure exposure of the hook exception later proves unacceptable, the reversal path is a content policy on hook-payload lines (redaction/short retention), not silently dropping the record.

We rejected, for the hook case, a separate opt-in ephemeral `.debug/` channel (a hook that is off by default is off exactly when the unexpected failure you needed it for occurs — the record must exist by default to be useful), and truncated/redacted hook content (unreliable and defeats the lossless purpose). We retained the original rejections for non-hook content: logging JSONL or PTY bodies (JSONL is already persisted; PTY is noise) and threading `runId`/`stage` for richer correlation (time-ordering already supplies it).
