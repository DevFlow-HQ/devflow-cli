# Graceful Completed-Session Cleanup for Structured Runners

The four structured provider runners (`claudeHookDrivenSessionRunner`, `claudeJsonlSessionRunner`, `codexHookDrivenSessionRunner`, `codexJsonlSessionRunner`) drive an *interactive* provider CLI over a PTY. When a stage's **Completion marker** arrives on the structured **Data plane** and the session is finalized, the provider process is still sitting at its prompt — it has no reason to exit on its own. The runners previously only **armed a passive cleanup timer**: they waited `cleanupTimeoutMs` (5s) for a natural PTY exit and then rejected with `ProviderSessionCleanupError`. In practice the provider never exits unprompted, so a valid, fully-completed stage routinely failed on cleanup timeout. Their failure paths compounded this by calling raw `harness.kill()` without first giving the provider a chance to tear down its own child processes.

The PTY-synthesis fallback runner (`ptyManagedSessionRunner`) already solved the happy-path half of this with an adapter-supplied `cleanupCommand` (`/exit\n`): on valid completion it *writes* the command so the provider exits itself. We port and extend that idea to the structured runners, and we make the policy a first-class capability of the shared PTY-control boundary rather than four near-duplicate copies.

**Decision.** `ptyControlHarness` gains a `shutdown({ command?, timeoutMs })` method: if the PTY is still alive it writes the provider-native graceful-exit command (raw, newline-terminated — *not* bracketed-paste, so a slash command is interpreted as a command), waits up to `timeoutMs` for exit, then force-kills. It resolves once the PTY is dead. The harness keeps an internal exit latch independent of the disposable handler subscription so `shutdown` still observes a natural exit after `dispose()` has restored terminal raw-mode. Each structured runner command carries a `cleanupCommand`, set by the adapter: Claude → `/exit\n`, Codex → `/quit\r` (carriage return = Enter; a `\n` only inserts a newline at the Codex prompt rather than submitting).

- **Valid completion:** on finalize-before-exit the runner calls `harness.shutdown(...)` and, when it resolves, settles the session as **success** (synthesizing exit metadata, guarded so a racing `onExit` cannot double-settle). A force-kill is teardown, not failure — the marker already matched and the stage already validated.
- **Failure paths:** if the PTY is still alive the runner fires a *detached* `harness.shutdown(...)` (graceful write + force-kill backstop) and rejects with the original error immediately; teardown never delays error surfacing. If the PTY already exited, it rejects straight away.
- `cleanupTimeoutMs` is **repurposed** from "wait for natural exit, then error" to "wait after graceful exit, then force-kill." Default stays 5s.
- `ProviderSessionCleanupError` is **narrowed** to the single case where `kill()` itself throws.

**Trade-offs.** A force-kill now counts as a *successful* outcome, and `cleanupTimeoutMs` means the opposite of what it used to — both surprising without this context, hence this record. We accept that a provider which ignores its graceful-exit command is still reported as success: the completion marker is the **single authoritative done signal**, and a clean process exit is teardown hygiene, not stage correctness. The graceful-exit strings are provider knowledge living in the adapters (consistent with the fallback `cleanupCommand`); a wrong string degrades to force-kill rather than breaking. Codex's `/quit\r` was verified against a real Codex CLI (carriage return, not `\n`); the force-kill backstop makes any future drift non-fatal regardless. The provider event data planes, marker finalization, validation/repair, transcript capture, and the JSONL post-exit drain are deliberately unchanged.
