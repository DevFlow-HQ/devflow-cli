# Provider-Authored Issue Files With Existence-Only Validation

The issue-decomposition stage should let the selected provider write issue markdown files directly into the run's issues directory (`.devflow/runs/<run-id>/issues/`), and DevFlow should validate only that at least one non-empty issue file exists before completing the stage. DevFlow hands the provider the issues directory path through the prompt and does not author, name, rename, or arbitrate issues — consistent with DevFlow's role as a delegator/orchestrator rather than an owner of issue identity. This means the existing `run.writeIssue(slug, content)` helper (with its slug normalization and duplicate-slug rejection) is intentionally bypassed by this stage: it cannot coexist with direct provider writes because run artifacts are written with an immutable `wx` flag, and forcing issue content back through DevFlow would require a structured multi-issue output channel that the narrow provider-event design deliberately avoids. We rejected devflow-mediated writes and stricter validation (filename/slug well-formedness, required-heading structure, blocked-by cross-reference checks) because issues are authored and later consumed by providers, no execute stage consumes their structure yet, and stricter checks would invite flaky repair loops; tighten validation when a real consumer requires it. The stage reads the canonical PRD as its contract in a dedicated session, gets one in-session repair attempt and the standard provider-backed stage retry boundary (which clears the issues directory before re-attempting so leftover files cannot falsely satisfy the existence check), and relies solely on the standard completion marker plus the directory scan — no additional sentinel.
