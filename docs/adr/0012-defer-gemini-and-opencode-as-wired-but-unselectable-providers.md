# Defer Gemini And OpenCode As Wired-But-Unselectable Providers

DevFlow's MVP supported provider set is **Claude** and **Codex** — the providers with a working structured data plane (hooks/JSONL) that truthfully satisfies the normalized provider-event contract. **Gemini** and **OpenCode** are *deferred*: their adapters and the hooks→JSONL→PTY fallback-tier infrastructure stay fully wired and test-covered, but they are excluded from *every* CLI selection path — the first-run picker, single-installed auto-select, the explicit `--provider` flag, and saved-config defaults all treat them as not selectable — so a user can never reach them from the CLI (only tests can, against the adapter factory). They are deferred because today they would run PTY-synthesis-only, which cannot deliver clean per-turn content, a reliable provider session id, or resume; shipping them as selectable would be a *broken*, not merely rough, experience.

We rejected **deleting** the adapters (they are the load-bearing investment behind ADR-0002's fallback-tier architecture, and a deferred provider is promoted to supported by flipping a membership list and wiring its event source — deleting now just means rewriting later) and rejected **shipping them as "experimental"** (a label is a fig leaf over a broken experience, and it invites users into confusing failure modes). We also chose to keep deferred providers **fully invisible** rather than surfacing a "coming soon" hint, because promotion depends on the upstream provider's maker shipping hooks/JSONL/other structured output DevFlow can build on — an open-ended dependency DevFlow cannot promise against. "Deferred" is distinct from an *unsupported* provider id (an unrecognized name such as a typo), which is not a built-in at all.

## Consequences

- The user-facing "supported MVP claims" (README, error strings that enumerate supported providers) name only Claude and Codex.
- A previously-saved `defaultProvider` of a deferred provider is rejected on the saved-config path like any other non-selectable selection, guiding the user to re-pick a supported provider.
- The supported set is a single membership list (`SUPPORTED_PROVIDER_IDS`), kept deliberately separate from the wired set (`BUILT_IN_PROVIDERS`): the former means "users may select this", the latter means "an adapter exists for this". Discovery, all four selection paths (picker, single-installed auto-select, `--provider`, saved-config default), and the "supported providers" error enumerations all *derive* from the membership list — nothing hard-codes the supported set elsewhere.
- This makes deferral and restoration symmetric one-line edits with no ripple. **Demotion** — when a provider ships a breaking change that corrupts orchestration — is removing its id from the membership list; it instantly disappears from every selection path while its adapter stays wired and test-covered. **Promotion** is adding the id back (and, for a brand-new provider, wiring its structured event source). No structural change to selection, discovery, or the fallback tiers in either direction.
