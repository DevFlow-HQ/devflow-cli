# Bind the Hook IPC Socket at a Short, Machine-Scoped Path

The **Provider hook** hands its payload back to DevFlow over a Unix domain socket. We originally placed that socket inside the **Scoped provider home** (`.devflow/runs/<run-id>/.claude/devflow-hooks/hook.sock`, `…/.codex/hook.sock`), co-located with every other run artifact. On macOS/BSD `sockaddr_un.sun_path` is capped at **104 bytes** (vs 108 on Linux), and `listen()` silently *truncates* an over-length path instead of erroring — so the run-scoped path silently failed to bind for any moderately nested repo (Claude's suffix alone is 59 bytes, so any repo root over ~44 bytes broke it). Because `resolveProjectRoot` returns git's canonical `/private/...` toplevel on macOS, real installs hit this routinely.

We bind the socket at a short, machine-scoped path instead: `<os.tmpdir()>/devflow-<hash>.sock`, where `<hash>` is a fixed-length digest of `workingDirectory + phase.id` (computed by a shared `resolveHookSocketPath` used by both runners and tests). This keeps the path length bounded regardless of repo depth or run-id, and unique across concurrent runs and repos. The hook discovers it through the injected `DEVFLOW_HOOK_IPC_PATH` env var, so relocation is transparent to the hook script; `hook.js` and provider config stay in the run tree. As defense in depth, `hookSocketServer.start()` rejects with a typed `HookSocketPathTooLongError` (platform-aware budget, overridable for tests) before `listen()` rather than letting the OS truncate.

The trade-off: the socket is deliberately **not** part of the Scoped provider home, so it is no longer co-located with the run's artifacts. We accept that loss of co-location — the socket is an ephemeral IPC endpoint, not durable run state — in exchange for a bind that is correct on macOS irrespective of where the repo lives.
